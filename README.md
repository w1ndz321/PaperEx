# PaperEx — 学术论文知识抽取流水线

PDF → Markdown → LLM 抽取结构化知识（10 类知识条目），支持并发 + 断点续跑。

全程状态由 SQLite 管理：早期版本为 `kg_pipeline.db`（根目录），当前版本为 `db/<batch>.db`，每个批次独立库。

---

## 流程概览

```
pdfs/<batch>/*.pdf
    ↓ kg_convert.py      PDF→MD，去重，过滤残缺文件
markdown/<batch>/*.md
    ↓ kg_preprocess.py   LLM 修正 metadata + 学科分类（1次LLM调用）
kg_output/<batch>/*.json  (只有 metadata)
    ↓ kg_extract.py      LLM 抽取 10 类知识（1次LLM调用）
kg_output/<batch>/*.json  最终产物：metadata + entries
```

## 硬件消耗 & 成本估算（1,000 万篇）

按批次 2 实测（1000 PDF / 128 线程 / 55.1% 有效率）外推。

### convert（PDF→MD）— CPU 密集，4 线程最优

pymupdf4llm 底层 C 库内部自带多核并行，Python 层线程数超过 4 后内部线程争抢 CPU，吞吐反而下降。

| 线程数 | 吞吐 | CPU使用率 | 单机日处理量 | 1,000 万篇耗时 |
|-------|------|-----------|------------|--------------|
| **4** | **35.1 篇/min** | ~21 核 | **~50,600 篇** | **~198 天** |
| 8 | 32.6 篇/min | ~22 核 | ~47,000 篇 | ~213 天 |
| 16 | 29.3 篇/min | ~22 核 | ~42,200 篇 | ~237 天 |

> 实测环境：Intel Xeon Platinum 8380 × 160 核，1007 GB 内存。测试集 100 篇 PDF。

### extract（LLM 知识抽取）— API 密集，按线程数扩展

瓶颈在 API 限流（非硬件），吞吐随线程数近线性增长。

| 方案 | 总线程 | 有效篇数 | 耗时 | 硬件消耗 |
|------|-------|---------|------|---------|
| 128 线程（1 台） | 128 | 551 万 | **~70 天** | CPU<1%, Mem<50MB |
| 256 线程（1 台） | 256 | 551 万 | **~35 天** | 同上 |
| 512 线程（1 台） | 512 | 551 万 | **~17.5 天** | 同上 |
| 1024 线程（1 台） | 1024 | 551 万 | **~8.8 天** | 同上 |

> preprocess 仅 53 秒即可完成 551 篇，相对可忽略，不单独列。

### 费用估算（DeepSeek-v3.2 定价：1 元/百万输入，2 元/百万输出）

| | Tokens | 费用 |
|--|--------|------|
| 输入（preprocess + extract） | ~432 亿（65%） | **4.32 万元** |
| 输出（preprocess + extract） | ~232 亿（35%） | **4.64 万元** |
| **合计** | **~664 亿** | **~8.96 万元** |

> 仅有效论文产生费用，跳过文件不计。实际受缓存命中率、批量折扣影响。

## 使用方法

```bash
python kg_convert.py --input-dir pdfs/<batch>     # PDF→MD + 去重过滤
python kg_preprocess.py markdown/<batch>           # LLM 修正元数据 + 学科分类
python kg_extract.py markdown/<batch>              # LLM 抽取 10 类知识条目
```

状态查看：`python kg_state.py`；重新处理：加 `--force`；调试：加 `--stream`（单线程）。

---

## 注意事项

1. **必须按序运行**：convert → preprocess → extract，extract 依赖 preprocess 的 metadata
2. **中断可续跑**：直接重跑即可，已完成自动跳过；running 超 10 分钟自动重置为 pending
3. **API 限流是唯一瓶颈**：硬件资源充足（CPU < 1%，内存增量 < 50MB），可安全提高 WORKERS

## 本阶段优化记录

- **Preprocess 增加 metadata 校验**：原 preprocess 仅做学科分类，后续加入 LLM 对 title/year/DOI 的二次修正，正则提取准确率受文章格式影响大，LLM 审查后显著提升
- **数据质量过滤**：MD5 内容去重（~5.5% 重复率） + 字符数过滤（<10K 字符跳过），跳过文件约 38-45%，节省无效 LLM 调用
- **稳定性**：JSON 解析改用 `raw_decode()` 防止 LLM 尾部多余文字导致失败；分级指数退避（429 限流 vs 网络错误 vs JSON 失败）；doc_id 改为 MD5(标题+文件名) 消除冲突
- **工程化**：SQLite 状态管理替代文件检查，支持原子性任务领取、断点续跑、超时重置；ThreadPoolExecutor 并发，LLM 调用本质是网络 I/O 等待，GIL 自动释放；分阶段 token 记录入库，支持实时成本追踪
